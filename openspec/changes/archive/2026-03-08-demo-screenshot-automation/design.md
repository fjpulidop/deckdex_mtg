## Implementation Design

### Script: `frontend/scripts/screenshot-demo.mjs`

The script is a standalone Node ESM module (`.mjs`) that uses Playwright's programmatic
API directly — it does NOT run through `playwright test` or use `playwright.config.ts`.
This is intentional: `playwright.config.ts` is wired for the `webServer` auto-start
feature, which would conflict if the dev server is already running. A standalone script
gives the developer explicit control.

**Why ESM (.mjs) and not CJS (.js)?**
`frontend/package.json` declares `"type": "module"`, making `.js` files ESM by default.
Using `.mjs` is the idiomatic extension and is consistent with how other scripts in this
project are named (`debug-demo.mjs`).

**Why `chromium.launch()` and not `chromium.launchServer()`?**
Single-use script; no need for a persistent browser server.

**Resolution: 1280×800**
Matches the spec requirement. The `Hero.tsx` image container is `aspect-[3/2]` (which is
1280×853 at full width), so 1280×800 captures the card table with a realistic viewport
showing the demo banner and first ~10-12 card rows.

**Wait strategy:**
1. `waitUntil: 'networkidle'` on navigation — ensures React hydration is complete.
2. `waitForSelector('table tbody tr', { timeout: 10_000 })` — waits for at least one
   rendered card row. Demo data is synchronous so this resolves in <500 ms.
3. `waitForTimeout(500)` — extra buffer for CSS transitions (the framer-motion fade-in
   on the demo banner).

**Output path:**
`resolve(__dirname, '../public/dashboard-preview.png')` — resolves to
`frontend/public/dashboard-preview.png` regardless of where `node` is invoked from,
because `__dirname` is derived from `import.meta.url`.

**Error handling:**
The script uses top-level `await` (valid in Node 14.8+ with ESM). Unhandled rejections
terminate the process with a non-zero exit code, which surfaces naturally in the terminal
and in any CI context that might wrap this script.

### npm script: `package.json`

```json
"screenshot:demo": "node scripts/screenshot-demo.mjs"
```

Run from the `frontend/` directory: `npm run screenshot:demo`.

**Prerequisite:** The dev server must be running (`npm run dev` in a separate terminal).
The script does not auto-start it, keeping the surface area minimal and avoiding port
conflicts with any running server.

**Playwright browser prerequisite:** `@playwright/test` is already a devDependency.
Chromium must be installed separately: `npx playwright install chromium`. This is a
one-time step per machine. The script will fail fast with a clear Playwright error if
the browser is not installed.

### Output: `frontend/public/dashboard-preview.png`

Committed to the repository as the initial baseline. The file is served by Vite's static
asset server at `/dashboard-preview.png` (no path prefix). `Hero.tsx` references it as
`src="/dashboard-preview.png"`.

The file should be regenerated by a developer whenever the Dashboard or Demo UI changes
significantly. There is currently no automated trigger for this — it is a deliberate
manual step (see Non-goals).

## Architectural Decisions

### Decision 1: Standalone script vs. Playwright test

**Options considered:**
- A) Playwright test spec in `frontend/e2e/` that uses `page.screenshot()`.
- B) Standalone Node script in `frontend/scripts/`.

**Decision: B (standalone script).**

Reasoning: A Playwright test spec runs through `npx playwright test`, which (per
`playwright.config.ts`) would start a second Vite dev server on port 5173 if one is not
running. The `webServer.reuseExistingServer: true` flag helps but introduces ordering
sensitivity. More importantly, a screenshot generation tool is not a test — it has no
`expect()` assertions and does not belong in the test suite. Keeping it in `scripts/`
makes its purpose clear and avoids polluting test reports.

### Decision 2: Commit the PNG vs. .gitignore it

**Options considered:**
- A) Commit `dashboard-preview.png` to the repository.
- B) `.gitignore` it and require each developer to regenerate.

**Decision: A (commit it).**

Reasoning: The landing page hero degrades gracefully when the image is missing (the
`onError` handler in `Hero.tsx` hides the `<img>` and shows a gradient fallback), but
the product experience is significantly worse without it. Committing the baseline image
ensures the landing page looks correct immediately after `git clone` without requiring
an extra step. Regeneration is done only when UI changes warrant it.

### Decision 3: `networkidle` vs. `load` wait strategy

`waitUntil: 'networkidle'` is stricter than `'load'` — it waits for no network activity
for 500ms. Since `/demo` makes zero API calls (all data is static mock data), `networkidle`
is reached quickly and is the safest signal that React has finished rendering.

## Risks and Edge Cases

| Risk | Likelihood | Mitigation |
|---|---|---|
| Chromium not installed on new dev machine | Medium | Error message from Playwright is self-explaining; document `npx playwright install chromium` in task notes |
| Dev server not running when script is invoked | High | Script will fail with a connection-refused error — expected and acceptable behaviour |
| Demo banner covers card table partially | Low | 500ms buffer after table is visible gives CSS transitions time to settle |
| Image file size too large for git | Low | PNG at 1280×800 for a mostly-static UI compresses to ~70-100 KB — acceptable |
| Playwright API changes breaking the script | Very Low | `@playwright/test ^1.56.0` is pinned in devDependencies |

## Data Flow

```
Developer terminal
    └── npm run screenshot:demo (from frontend/)
            └── node scripts/screenshot-demo.mjs
                    └── chromium.launch() [headless]
                            └── page.goto('http://localhost:5173/demo')
                                    └── Vite dev server (must be running)
                                            └── Demo.tsx + DemoContext + Dashboard
                                                    └── demoData.ts (35 mock cards, no API calls)
                            └── waitForSelector('table tbody tr')
                            └── page.screenshot({ path: 'frontend/public/dashboard-preview.png' })
                    └── browser.close()
                    └── console.log("Screenshot saved to ...")
```
